1. What Does This Framework Do?

This framework automates testing of the REX Export Permit system — the government system that Australian exporters use to lodge, amend, and replace export certificates for commodities like horticulture, grain, and meat.

Key Capabilities

  • SOAP API Testing — Sends real SOAP requests (ORDER, LODGE, AMEND, REPLACE, READ) to the REX services and validates responses
  • Staff Portal UI Testing — Automates the staff portal login, inspection, and authorisation workflows
  • Hybrid Flows — Combines SOAP + UI in a single test (e.g. LODGE via API → authorise via portal → REPLACE via API)
  • Business Rule Validation — Tests specific rules like departure date limits, certificate requirements
  • Multi-Environment — Runs against SIT, SIT2, or VND with a single command
  • Multi-Commodity — Supports Horticulture, Grain, and Meat with shared builder logic

Stage 1 — Defaults + Overrides

Each commodity file defines its DEFAULTS (product type, pack type, destination country, etc.). Tests can override any field:

SourceExample Values

Stage 2 — Builder Factory

The createCommodityBuilders(DEFAULTS) factory produces 6 builder functions. Each commodity file just calls the factory once — zero duplicated builder code.

Builder FunctionReturnsUsed For
buildOrderPayload()OrderRexPayloadMinimal ORDER (country + product)
buildLodgePayload()LodgeRexPayloadMinimal LODGE (country + product)
buildDefaultOrderPayload()OrderRexPayloadFull ORDER with all defaults
buildDefaultLodgePayload()LodgeRexPayloadFull LODGE with all defaults
buildDefaultAmendPayload()AmendRexPayloadFull AMEND (requires rexState)
buildDefaultReplacePayload()ReplaceCertificatePayloadFull REPLACE (requires rexState)

To Add a New Commodity

  1. Create test-data/commodities/new-commodity.ts
  2. Define COMMODITY_TYPE, PRODUCT_TYPE, and DEFAULTS
  3. Call createCommodityBuilders(DEFAULTS) and export the destructured builders
  4. Done — all 6 builder functions are instantly available

How Tests Use Page Objects

// Individual page object calls:
await loginPage.login(username, password);
await rexSearchPage.searchByRexNumber(rexNumber);
await rexDetailPage.addInspectionDetails({ startDate, endDate });
await rexDetailPage.authorise('Authorised');
await rexDetailPage.assertREXStatus('CertificateReady');

Reusable Workflow Fixture — authoriseRex

Since login → search → inspect → authorise is repeated across many tests, a fixture wraps all four steps. Tests just pass the REX number — no page objects needed:

// Destructure from test fixtures — just like soapClient:
test('my test', async ({ soapClient, authoriseRex }) => {

  // One line replaces four repeated steps:
  await authoriseRex(rexNumber);

  // Or with custom options:
  await authoriseRex(rexNumber, {
    inspection: { inspectorName: 'Inspector Smith' },
    authoriseComments: 'Approved',
  });
